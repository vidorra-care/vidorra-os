01 — 市场定位

竞品分析
与差异化机会

Web Desktop OS 并不是空白市场，但现有产品都在关键维度上存在明显短板。理解这些短板，就是你的机会窗口。

项目 macOS 风格 框架无关 App 跨 App 数据层 应用商店 活跃维护 自托管

核心差异化：没有任何现有项目同时实现了 ① macOS 级视觉审美 ② 真正框架无关的 App 沙箱（你的 App 可以是 Vue、Svelte、Next.js、甚至纯 HTML） ③ 结构化跨应用数据层。这三者的组合是你的护城河。

02 — 产品哲学

设计原则

WebOS 不是一个应用，它是一个让其他应用存在的世界。

六条核心原则

🏛
底座即平台
内核只提供系统能力，不预设应用形态。博客、记账、天气——都由第三方 App 实现。
🔒
沙箱优先
每个 App 运行在独立 iframe 沙箱内，通过受控 postMessage 协议与内核通信，互不污染。
🌐
框架无关
App 可以用任何技术栈构建——React、Vue、Svelte、Angular，甚至原生 HTML。SDK 是协议，不是框架。
📦
数据互通
打通应用之间的数据孤岛——记账 App 的分类数据可以被统计 App 直接读取，前提是授权。
🎨
macOS 审美
玻璃质感、精确的阴影、流畅的动画——用户不需要知道它是 Web 的。
🔌
渐进增强
最简 App 只需一个 URL。复杂 App 可以深度接入数据层、通知、快捷键、菜单栏定制。
03 — 系统架构

分层架构设计

系统由四层构成：内核层提供操作系统基础能力；Shell 层是 macOS 风格的 UI 外壳；Bridge 层是 App 与内核通信的消息总线；App 层是完全沙箱化的第三方应用。

APP 层
博客 App (Next.js) 记账 App (Vue) 天气 App (Svelte) 笔记 App (纯 HTML) … 任意 App
沙箱层
iframe sandbox postMessage 协议 权限声明 (manifest.json) origin 白名单
Bridge 层
KernelBus (消息总线) Permission Guard DataStore API EventEmitter
Shell 层
WindowManager Dock Menubar Spotlight Wallpaper Notifications ContextMenu
Kernel 层
AppRegistry DataStore (IndexedDB) AuthStore ThemeEngine ShortcutManager FileSystem (VFS)

关键设计决策：为什么用 iframe

iframe 沙箱是实现框架无关 App 的唯一可靠方式。每个 App 是独立的浏览器上下文，有自己的 JS 运行时、CSS 作用域、localStorage。

✓ 推荐方案
iframe + postMessage 协议
完全隔离、框架无关、安全、支持跨域 App；类似 VS Code 插件机制、Figma Widget 机制。成熟可靠。
✗ 不推荐
Module Federation / 直接 mount DOM
强制 App 使用相同框架版本，CSS 污染无法解决，安全边界模糊，牺牲了框架无关性。

参考先例：VS Code 扩展在独立的 Extension Host 进程中运行，通过 RPC 通信；Figma 的 Plugin 运行在 iframe + sandbox 中；Linear 的 Apps 也是类似机制。这个模式在工业界已被充分验证。

04 — 内核模块

内核模块设计

WindowManager（窗口管理器）

负责所有窗口的生命周期、层级（z-index）、拖拽、缩放、最小化/最大化、窗口快照（Mission Control）。

// 窗口描述符 — 由 AppRegistry 在打开 App 时创建
interface WindowDescriptor {
  id: string
  appId: string
  title: string
  url: string           // App 的实际 URL，加载到 iframe.src
  icon: string          // SVG / URL
  rect: WindowRect
  state: 'normal' | 'minimized' | 'maximized' | 'fullscreen'
  focused: boolean
  zIndex: number
  // 传给 iframe sandbox 属性的权限集
  sandboxFlags: SandboxFlag[]
}

interface WindowRect {
  x: number; y: number
  width: number; height: number
  minWidth: number; minHeight: number
}

AppRegistry（应用注册表）

管理已安装的 App，读取并校验每个 App 的 manifest.json，提供安装/卸载/启动/停止接口。

// manifest.json — 每个 App 必须提供
{
  "id": "com.yourname.budget",           // 反向域名唯一 ID
  "name": "记账本",
  "version": "1.2.0",
  "entry": "https://budget.yourapp.dev",   // 或相对路径
  "icon": "./icon.svg",
  "category": "finance",
  "defaultSize": { "width": 900, "height": 600 },
  "minSize": { "width": 400, "height": 300 },

  // 权限声明 — 用户安装时弹窗确认
  "permissions": [
    "datastore.read:com.yourname.budget.*",
    "datastore.write:com.yourname.budget.*",
    "datastore.read:com.global.categories",  // 读全局分类
    "notifications.send",
    "filesystem.read"
  ],

  // 注册到 Spotlight 的搜索命令
  "spotlightActions": [
    { "keyword": "新增收支", "action": "open?modal=new-entry" }
  ],

  // 菜单栏定义（当本 App 聚焦时显示）
  "menubar": {
    "文件": [
      { "label": "新建记录", "action": "new-entry", "shortcut": "CmdOrCtrl+N" },
      { "type": "separator" },
      { "label": "导出 CSV", "action": "export-csv" }
    ]
  },

  // 向其他 App 暴露的数据 Schema
  "exposes": {
    "transactions": {
      "description": "用户的所有收支记录",
      "schema": "./schemas/transaction.json"
    }
  }
}

DataStore（跨应用数据层）

基于 IndexedDB 的结构化数据存储，是整个系统最具创新性的模块。详见 §06。

ShortcutManager（快捷键管理器）

全局快捷键注册中心。App 通过 manifest 声明自己的快捷键，当 App 聚焦时，ShortcutManager 将键盘事件路由到 App 的 iframe 内。

ThemeEngine（主题引擎）

管理系统级 CSS 变量（颜色、圆角、字体、模糊量）。App 可通过 SDK 订阅主题变化，实现 Dark Mode 无缝切换。

05 — App SDK

SDK规范

SDK 本质上是一个轻量级的 postMessage 封装库，提供 Promise 化的 API。App 引入 SDK 后，可以像调用本地函数一样调用系统能力。SDK 本身不超过 8KB gzip。

设计理念：SDK 是协议适配器，不是框架。用 React 的 App 可以用，用 Vue 的也可以用，甚至纯 HTML 脚本引入即可使用。

安装与初始化

// npm install @webos/sdk
import { createApp } from '@webos/sdk'

const app = createApp()

// 等待内核握手完成后才能调用 API
await app.ready()

// 或 CDN 引入（框架无关场景）
// <script src="https://cdn.webos.dev/sdk.min.js"></script>
// const app = WebOS.createApp()

窗口控制

call app.window.setTitle(title) 
app.window.setTitle('记账本 — 2024年6月')

// 设置窗口尺寸
app.window.resize({ width: 1200, height: 800 })

// 请求全屏
await app.window.requestFullscreen()

// 创建子窗口（modal 或独立窗口）
const childWin = await app.window.createChild({
  url: '/settings',
  title: '偏好设置',
  modal: true,
  size: { width: 560, height: 400 }
})

通知

call app.notifications.send(options) 
await app.notifications.send({
  title: '月度账单已生成',
  body: '2024年6月支出 ¥3,240，点击查看明细',
  icon: './icon.svg',
  onClick: () => app.window.focus()
})

菜单栏响应

event app.on('menu:action', handler) 
// 接收来自菜单栏的 action（由 manifest.menubar 定义）
app.on('menu:action', ({ action }) => {
  switch (action) {
    case 'new-entry': openNewEntryModal(); break
    case 'export-csv': exportData(); break
  }
})

Spotlight 集成

event app.on('spotlight:action', handler) 
// 处理来自 Spotlight 搜索的 deep link action
app.on('spotlight:action', ({ action, params }) => {
  if (action === 'open?modal=new-entry') {
    openNewEntryModal()
  }
})

主题订阅

event app.theme.subscribe(handler) 
// 订阅系统主题变化（浅色/深色/accent 颜色等）
app.theme.subscribe(({ mode, tokens }) => {
  document.documentElement.dataset.theme = mode  // 'light' | 'dark'
  // tokens 包含内核暴露的 CSS 变量值
})

// 也可以直接获取当前主题
const theme = await app.theme.get()
06 — 跨应用数据层

数据互通设计

记账 App 的数据可以被图表 App 直接读取——这才是真正打通数据孤岛的"操作系统"。

这是整个产品最有价值的差异化特性。DataStore 是一个结构化的、Schema 驱动的、具有权限控制的跨 App 数据总线。

命名空间设计

所有数据按命名空间组织。格式为 appId:collection，类似 k8s 的资源组织方式。

// 私有数据 — 只有 com.yourname.budget 自己可写
"com.yourname.budget:transactions"

// 全局数据 — 任何有权限的 App 可读写
"com.global:categories"
"com.global:contacts"
"com.global:events"

DataStore API

call app.data.collection(ns).query(filter) 
// 查询记录（需要在 manifest 声明 datastore.read 权限）
const transactions = await app.data
  .collection('com.yourname.budget:transactions')
  .query({
    where: { month: '2024-06', category: '餐饮' },
    orderBy: [{ field: 'date', dir: 'desc' }],
    limit: 50
  })

// 写入记录
await app.data
  .collection('com.yourname.budget:transactions')
  .insert({
    id: uuid(),
    amount: -128.5,
    category: '餐饮',
    note: '午饭',
    date: '2024-06-15'
  })

// 实时订阅（类似 Firestore onSnapshot）
const unsub = app.data
  .collection('com.yourname.budget:transactions')
  .subscribe(records => {
    console.log('数据更新', records)
  })

权限流程

安装时
读取 manifest.permissions → 向用户展示权限弹窗 → 用户确认/拒绝 → 权限写入 AuthStore
运行时
App 发起 data.query 请求 → KernelBus 拦截 → Permission Guard 校验 → 通过：返回数据 / 拒绝：抛出 403

跨 App 数据流示例

// ──── 场景：统计图表 App 读取记账 App 的数据 ────

// 图表 App 的 manifest.permissions 声明：
"permissions": [
  "datastore.read:com.yourname.budget:transactions"
]

// 图表 App 在运行时直接查询记账数据（无需 API 对接）：
const data = await app.data
  .collection('com.yourname.budget:transactions')
  .query({ where: { year: 2024 } })

const byMonth = groupBy(data, 'month')
// → 直接渲染为折线图，无需记账 App 提供任何后端接口

这是真正的数据互通：博客 App 的文章数据可以被 SEO 分析 App 读取；联系人 App 的数据可以被邮件 App 读取；健身记录可以被健康分析 App 读取。数据属于用户，存在 IndexedDB 中，App 只是视图。

07 — 应用商店

App Store设计

应用商店本身也是一个 App（id: com.webos.appstore），通过读取一个中央 Registry JSON 来展示可用的 App 列表。这个 Registry 可以是 GitHub 仓库中的一个 JSON 文件。

App 发布流程

开发者
开发 App（任意技术栈） → 编写 manifest.json → 部署到任意 URL → 提交 PR 到 Registry 仓库
审核
自动校验 manifest schema 人工审核（可选） 合并 PR → App 上架
用户安装
App Store 展示 App → 用户点击安装 → 权限弹窗确认 → 写入 AppRegistry → 出现在 Dock / 桌面

Registry 数据结构

// registry/apps.json（托管在 GitHub）
{
  "apps": [
    {
      "id": "com.example.blog",
      "name": "轻博客",
      "description": "Markdown 博客写作与发布",
      "author": "Jane Doe",
      "icon": "https://blog.example.com/icon.svg",
      "manifestUrl": "https://blog.example.com/manifest.json",
      "category": "productivity",
      "tags": ["blog", "writing", "markdown"],
      "screenshots": ["..."],
      "featured": true
    }
  ]
}

私有部署支持

企业或个人可以指定自己的 Registry URL，从而建立私有应用商店，不依赖公共 Registry。这对于内部工具场景（类似你之前展示的 元镜OS 那样的企业数据平台）非常有价值。

08 — 技术选型

技术选型

内核与 Shell（宿主侧）

模块选型理由
UI 框架 React 18 + TypeScript 生态最大，Concurrent Mode 对窗口管理有利，易于招募贡献者
状态管理 Zustand 轻量、无样板代码、React 无关（内核逻辑可独立于 React）
数据持久化 Dexie.js (IndexedDB) 比原生 IndexedDB API 友好，支持 reactive query，离线优先
进程间通信 自研 KernelBus 封装 postMessage，双向 RPC，带 request/response 模型和超时处理
动画 Framer Motion 窗口动画、Dock 弹簧效果、过渡动画
快捷键 tinykeys 650B，支持 Mac 符号（⌘⌥⇧），处理 iframe 内外的键盘路由
构建工具 Vite + pnpm workspaces Monorepo 结构（kernel、shell、sdk、apps 分包）

SDK（App 侧）

模块选型理由
核心通信 原生 postMessage 无依赖，8KB 以内，任何框架可用
类型定义 TypeScript Declaration @webos/sdk 提供完整 .d.ts，自动补全友好
React 绑定 @webos/react（可选） useWindow()、useTheme()、useDataStore() 等 hooks，按需安装

09 — 路线图

分阶段路线图

P1
核心底座（MVP）
0 → 8 周
  • WindowManager：拖拽、缩放、层级、最小化/最大化、动画
  • iframe 沙箱容器（WindowFrame 组件）
  • KernelBus：postMessage RPC 封装，双向通信
  • AppRegistry：manifest 解析、本地安装/卸载
  • Shell：Menubar（带时钟）、Dock（带 magnification）、桌面图标
  • Spotlight：App 搜索、快速启动
  • 右键菜单、通知系统
  • SDK v0.1：window、notifications、menu:action
  • 内置示例 App：Settings、Finder（浏览 DataStore）
P2
数据层 + App Store
8 → 16 周
  • DataStore：IndexedDB + Dexie，命名空间隔离
  • Permission Guard：安装时授权弹窗，运行时鉴权
  • SDK 扩展：data.collection() API，实时订阅
  • App Store App：读取 registry/apps.json，安装/卸载流程
  • 虚拟文件系统（VFS）：App 间共享文件
  • ThemeEngine：CSS 变量系统，Dark Mode 支持
  • 官方示例 App 2 个（Notes、Timer）用于验证 SDK
P3
生态建设
16 → 28 周
  • 开发者文档网站（docusaurus）
  • @webos/create-app 脚手架（类似 create-react-app）
  • @webos/sdk-react：useWindow、useTheme、useDataStore hooks
  • Spotlight：支持 App 深度集成（自定义搜索结果）
  • Mission Control（多窗口概览）
  • 多账户支持（localStorage 隔离 / 后端存储）
  • 公开 Registry 仓库 + GitHub Actions 自动验证
P4
高级特性
28 周之后
  • 后端可选支持（Supabase / PocketBase 适配器），数据云同步
  • 分屏（Split View）
  • App 内 App（嵌套窗口）
  • Web Push 通知（Service Worker）
  • PWA 支持（可安装到桌面）
  • Plugin API（注入菜单栏扩展，类似 Chrome 扩展）
10 — 风险与对策

风险识别

技术风险

风险影响对策
iframe CSP 限制（部分网站拒绝被 iframe 加载） 中 App 必须自己部署，不是嵌套外部网站；文档明确说明；提供本地 dev server 模板
postMessage 性能（高频调用） 低 数据层用批量读取；实时订阅做 debounce；DataStore 在宿主侧，不需要跨 iframe
IndexedDB 存储上限（Safari 约 1GB） 中 DataStore 做存储配额管理；大文件走 VFS + Blob 分片存储
Safari 的 iframe 限制（第三方 Cookie/Storage 访问） 中 App 的 Storage 需求通过 DataStore API 满足，不依赖 App 自身的 localStorage

产品风险

风险影响对策
开发者嫌 SDK 太复杂，不愿开发 App 高 最简 App 只需一个 URL + manifest.json，零 SDK 依赖也可运行；SDK 是增强，不是必须
竞品（Puter）先发优势 中 差异化聚焦 macOS 审美 + 结构化数据层；目标不同的用户群体（开发者个人工具场景）
维护者精力分散 高 MVP 范围严格控制在 P1；设计文档开源，吸引贡献者；用 GitHub Projects 公开路线图

建议的第一步：用你现有的 元镜 OS 代码作为视觉原型，从 WindowManager + KernelBus + iframe 沙箱开始实现，用一个最简的 "Hello World" App 验证通信机制。先跑通 App → SDK → postMessage → KernelBus → WindowManager 这条链路，再展开其他模块。
