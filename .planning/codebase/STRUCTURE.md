## Directory Purposes

**packages/kernel:**
- Purpose: Pure TypeScript business logic layer, zero UI dependencies, can run independently
- Contains: Service classes (AppRegistry, ThemeEngine), type exports for Kernel layer
- Key files: `src/app-registry.ts`, `src/theme-engine.ts`, `src/app-registry.test.ts`, `src/theme-engine.test.ts`
- Dependencies: Only `@vidorra/types`; uses happy-dom for testing

**packages/shell:**
- Purpose: React-based user interface, orchestrates Kernel services, manages window/app lifecycle
- Contains: React components, Vite build configuration, main entry point
- Key files: `src/main.tsx` (React root), `src/App.tsx` (root component, currently placeholder)
- Dependencies: React 18.3.0, `@vidorra/kernel`, `@vidorra/types`; Vite for dev/build

**packages/sdk:**
- Purpose: Client library for apps to communicate with Kernel via KernelBus
- Contains: createApp() function (planned), KernelBusClient (planned)
- Key files: `src/index.ts` (currently stub with phase marker comment)
- Dependencies: Only `@vidorra/types`; zero external dependencies

**packages/types:**
- Purpose: Shared TypeScript interfaces consumed by all other packages
- Contains: AppManifest, KernelBusMessage, KernelBusResponse, WindowDescriptor, MenuItem, SpotlightAction
- Key files: `src/index.ts` (exports all), `src/manifest.ts`, `src/kernel-bus.ts`, `src/window.ts`
- Dependencies: None; pure type definitions

**apps/{app-name}:**
- Purpose: Built-in applications running in iframe sandbox, can use any tech stack
- Contains: Framework-specific code (React in current apps), build config, public assets
- Examples: `apps/app-store/`, `apps/settings/`, `apps/calculator/`
- Dependencies: `@vidorra/sdk`, `@vidorra/types`, React (or framework of choice)

**docs/adr/:**
- Purpose: Architecture Decision Records documenting design rationale
- Key files: `0001-monorepo-structure.md`, `0002-iframe-sandboxing.md`, `0003-postmessage-rpc.md`
- Status: Historical decisions; drives implementation direction

**docs/architecture/:**
- Purpose: System design documentation with data flow and layer definitions
- Key files: `layers.md` (detailed layer responsibilities), `data-flow.md` (message sequences)

**registry/:**
- Purpose: Application registry metadata for built-in and installed apps
- Key files: `built-in-apps.json` (currently empty array; populated at app install time)

**.planning/codebase/:**
- Purpose: GSD (Goal-Seeking Dispatch) planning artifacts generated by code analysis
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, CONCERNS.md
- These are consumed by `/gsd:plan-phase` and `/gsd:execute-phase` to guide implementation

## Key File Locations

**Entry Points:**
- `packages/shell/src/main.tsx`: React app root; mounts to `#root` DOM element
- `packages/shell/vite.config.ts`: Dev server runs on port 3000, production build
- `apps/{app-name}/src/main.tsx`: App entry point, rendered in iframe context

**Configuration:**
- `tsconfig.json`: Root TypeScript configuration (strict: true, ES2022 target, React JSX)
- `pnpm-workspace.yaml`: Defines workspace glob patterns for monorepo
- `vitest.config.ts`: Test runner configuration
- `.prettierrc`: Code formatting (indentation, quotes, trailing commas)
- `eslint.config.js`: Linting rules

**Core Logic:**
- `packages/kernel/src/app-registry.ts`: Manages app install/uninstall, manifest validation
- `packages/kernel/src/theme-engine.ts`: Theme state, CSS variable injection, dark/light/auto modes
- `packages/types/src/manifest.ts`: AppManifest interface defining app contract
- `packages/types/src/kernel-bus.ts`: KernelBusMessage and KernelBusResponse protocols

**Testing:**
- `packages/kernel/src/app-registry.test.ts`: Unit tests for AppRegistry
- `packages/kernel/src/theme-engine.test.ts`: Unit tests for ThemeEngine
- `packages/kernel/vitest.config.ts`: Kernel-specific test configuration

## Naming Conventions

**Files:**
- PascalCase for React component files: `App.tsx`, `WindowManager.tsx`
- camelCase for utility/service files: `app-registry.ts`, `theme-engine.ts`
- camelCase with `.test.ts` suffix for test files: `app-registry.test.ts`
- `.module.css` for scoped styles: `App.module.css`
- `index.ts` for barrel exports from directories

**Directories:**
- kebab-case for package directories: `packages/kernel`, `packages/shell`, `apps/app-store`
- camelCase for source subdirectories: `src/`, `dist/`, `src/components/`, `src/utils/`
- `src/` is the canonical source location; `dist/` for build outputs

**Types and Interfaces:**
- PascalCase for interface names: `AppManifest`, `WindowDescriptor`, `KernelBusMessage`
- Exported from `packages/types/src/` and imported via `@vidorra/types`

**Functions and Variables:**
- camelCase for functions and variables: `getApp()`, `appRegistry`, `themeEngine`
- Singleton instances exported with lowercase names: `appRegistry`, `themeEngine`

## Where to Add New Code

**New Feature:**
- Primary code: `packages/kernel/src/` (if business logic) or `packages/shell/src/` (if UI)
- Tests: Corresponding `.test.ts` file in same directory as implementation
- Types: Add to `packages/types/src/` if interface is shared; local types otherwise

**New Component/Module:**
- Shell component: Create `packages/shell/src/components/{ComponentName}.tsx`
- Kernel service: Create `packages/kernel/src/{service-name}.ts` with class and singleton instance export
- Type definitions: Add to `packages/types/src/{domain}.ts` (e.g., `window.ts` for WindowDescriptor)

**New Built-in App:**
- Create `apps/{app-id}/` directory with `package.json`, `tsconfig.json`, `vite.config.ts`, `src/main.tsx`, `src/index.html`
- Declare manifest in `registry/built-in-apps.json` or serve dynamically
- Use `@vidorra/sdk` for system integration (optional; apps work without SDK)

**Utilities and Shared Logic:**
- Kernel utilities: `packages/kernel/src/utils/` (e.g., validation helpers)
- Shell utilities: `packages/shell/src/utils/` (e.g., CSS helpers)
- SDK helpers: `packages/sdk/src/` (once implementation begins)

**Cross-Package Dependencies:**
- Always declare in package `package.json` using `workspace:*` protocol
- Import types from `@vidorra/types`
- Kernel never depends on Shell; Shell depends on Kernel

## Special Directories

**node_modules:**
- Purpose: Installed dependencies managed by pnpm
- Generated: Yes (pnpm install)
- Committed: No (.gitignore excludes)

**dist/:**
- Purpose: TypeScript and Vite build outputs
- Generated: Yes (pnpm build, tsc -b)
- Committed: No (.gitignore excludes)

**.git/:**
- Purpose: Git version control metadata
- Generated: Yes (git init)
- Committed: Yes (system directory)

**.kilo/:**
- Purpose: Kilo CLI tool data and node_modules (external reference repo)
- Generated: Yes (tooling)
- Committed: Yes (tracking tool state)

**.reference/:**
- Purpose: Reference implementations and inspiration (macOS web clones, design systems)
- Generated: No (cloned references)
- Committed: Yes (for team reference)

**.planning/:**
- Purpose: GSD planning documents auto-generated by analysis
- Generated: Yes (by `/gsd:map-codebase`)
- Committed: Yes (planning artifacts preserved in repo)
